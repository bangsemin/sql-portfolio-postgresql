### 01. 주문 데이터 구조 확인

주문 분석 전에 `orders` 테이블의 컬럼 구성과 실제 데이터 형태를 확인한다.

[SQL 실습 열기 →](../02_projects/day02_order_product_analysis/01_orders_preview.sql)

#### 실행 목적

집계나 조인을 시작하기 전에 주문 데이터가 어떤 기준으로 저장되어 있는지 확인한다.
특히 주문 번호, 고객 번호, 상품 번호, 주문 수량, 주문 일자처럼 이후 분석 기준이 되는 컬럼을 파악한다.

#### 핵심 문법

* `SELECT *`

  * 테이블의 모든 컬럼을 조회한다.
  * 처음 보는 테이블의 구조를 탐색할 때 사용한다.

* `LIMIT 10`

  * 조회 결과를 앞의 10개 행으로 제한한다.
  * 전체 데이터를 조회하지 않고 표본 데이터로 구조를 빠르게 확인할 수 있다.

#### 확인할 항목

* 주문을 식별하는 컬럼은 무엇인가?
* 고객과 상품을 연결하는 컬럼은 무엇인가?
* 주문 수량과 주문 일자는 어떤 형식으로 저장되어 있는가?
* NULL 값 또는 비정상적인 값이 보이는가?

#### 실행 결과

`orders` 테이블의 주요 컬럼과 자료형을 확인했다.

| 컬럼명        | 자료형  | 역할                                 |
| ------------- | ------- | ------------------------------------ |
| `order_id`    | integer | 주문을 고유하게 식별하는 기본 키(PK) |
| `customer_id` | integer | 주문 고객을 식별하는 값              |
| `product_id`  | integer | 주문 상품을 식별하는 값              |
| `order_date`  | date    | 주문 일자                            |
| `quantity`    | integer | 주문 수량                            |

표본 조회 결과에서 주문 4건이 확인되었으며, 주문 일자와 수량이 정상적인 형식으로 저장되어 있었다.

#### 해석

* `order_id`는 각 주문을 구분하는 기본 키로 사용된다.
* `customer_id`는 이후 `customers` 테이블과 연결할 수 있는 기준값이다.
* `product_id`는 `products` 테이블과 연결할 수 있는 기준값이다.
* `orders` 테이블에는 상품 가격이 없으므로, 매출 분석을 하려면 `products` 테이블의 가격 정보와 조인해야 한다.
* 주문 수량은 `quantity` 컬럼으로 관리되며, 판매량과 매출 계산의 기준이 된다.

#### 추가 학습 메모

* `orders`는 거래 사실을 저장하는 테이블이며, 고객·상품의 상세 정보는 별도 테이블에서 관리하는 구조다.
* 이런 구조는 중복 데이터를 줄이고 데이터 변경을 쉽게 하기 위한 관계형 데이터베이스 설계 방식이다.
* `customer_id`, `product_id`는 다른 테이블의 기본 키와 연결되는 외래 키(FK) 역할을 한다.

### 03. 주문 데이터 품질 점검

분석 전에 주문 데이터의 누락값과 비정상 수량 여부를 확인한다.

[SQL 실습 열기 →](../02_projects/day02_order_product_analysis/03_orders_quality_check.sql)

#### 실행 목적

분석 결과의 신뢰도를 확보하기 위해 주문 식별자, 고객·상품 연결값, 주문 일자, 주문 수량에 누락값이 있는지 점검한다.
또한 주문 수량이 0 이하인 비정상 데이터가 존재하는지도 확인한다.

#### 핵심 문법

* `CASE WHEN`

  * 조건에 따라 다른 값을 반환한다.
  * 여기서는 문제가 있으면 `1`, 없으면 `0`으로 변환한다.

* `SUM(CASE WHEN ... THEN 1 ELSE 0 END)`

  * 조건을 만족하는 행의 개수를 계산하는 실무형 패턴이다.

* `IS NULL`

  * 값이 비어 있는지 확인한다.

* `quantity <= 0`

  * 주문 수량이 0 이하인 비정상 데이터를 확인한다.

#### 실행 결과

| 점검 항목                   | 결과 |
| --------------------------- | ---: |
| `order_id` NULL             |  0건 |
| `customer_id` NULL          |  0건 |
| `product_id` NULL           |  0건 |
| `order_date` NULL           |  0건 |
| `quantity` NULL             |  0건 |
| `quantity <= 0` 비정상 수량 |  0건 |

#### 해석

* 주문 테이블의 주요 컬럼에서 NULL 값은 발견되지 않았다.
* 주문 수량이 0 이하인 비정상 데이터도 존재하지 않았다.
* 현재 데이터는 기본적인 주문 분석과 매출 계산을 진행할 수 있는 상태다.

#### 추가 학습 메모

* `SUM(CASE WHEN 조건 THEN 1 ELSE 0 END)` 패턴은 조건에 맞는 행의 개수를 세는 실무형 점검 쿼리다.
* NULL 값이나 비정상 수량이 있으면 집계 결과와 매출 계산이 왜곡될 수 있으므로 분석 전에 먼저 확인해야 한다.

### 추가 점검: 중복 주문 ID 확인

주문을 고유하게 식별해야 하는 `order_id`가 중복으로 저장되어 있는지 확인한다.

#### 사용 쿼리 목적

`order_id`는 주문 테이블의 기본 키이므로 원칙적으로 중복될 수 없다.
만약 중복된 `order_id`가 조회된다면 주문 데이터의 식별 기준에 문제가 있는 것이다.

#### 핵심 문법

* `GROUP BY order_id`

  * 같은 `order_id`를 가진 행끼리 묶는다.

* `COUNT(*)`

  * 각 `order_id`가 몇 번 등장했는지 계산한다.

* `HAVING COUNT(*) > 1`

  * 그룹화 이후 중복으로 등장한 주문 번호만 필터링한다.

#### 실행 결과

중복된 `order_id`는 조회되지 않았다.

| 점검 항목    | 결과 |
| ------------ | ---: |
| 중복 주문 ID |  0건 |

#### 해석

* `order_id`는 주문을 고유하게 식별하는 값으로 정상 관리되고 있다.
* 중복 주문 ID가 없으므로 주문 건수 집계와 주문별 분석을 신뢰할 수 있다.
* 기본적인 NULL 점검, 비정상 수량 점검, 중복 주문 점검을 모두 통과했으므로 다음 단계에서 상품 데이터를 확인한다.

### 04. 상품 데이터 구조 확인

주문 데이터와 연결할 상품 테이블의 구조를 확인한다.

[SQL 실습 열기 →](../02_projects/day02_order_product_analysis/04_products_preview.sql)

#### 실행 목적

`orders` 테이블에는 상품 가격과 상품명이 없기 때문에, 매출 분석을 위해 `products` 테이블의 구조를 먼저 확인한다.
특히 `product_id`, 상품명, 카테고리, 가격 컬럼을 파악한다.

#### 핵심 문법

* `SELECT *`

  * 상품 테이블의 모든 컬럼을 조회한다.

* `ORDER BY product_id`

  * 상품 ID 기준으로 정렬하여 상품 데이터를 순서대로 확인한다.

#### 확인할 항목

* 상품을 식별하는 컬럼은 무엇인가?
* 상품명과 카테고리는 어떤 컬럼에 저장되어 있는가?
* 가격 정보는 어떤 컬럼에 저장되어 있는가?
* `orders.product_id`와 연결할 수 있는 기준 컬럼이 존재하는가?

#### 실행 결과

`products` 테이블의 주요 컬럼과 자료형을 확인했다.

| 컬럼명         | 자료형                 | 역할                                 |
| -------------- | ---------------------- | ------------------------------------ |
| `product_id`   | integer                | 상품을 고유하게 식별하는 기본 키(PK) |
| `product_name` | character varying(100) | 상품명                               |
| `category`     | character varying(50)  | 상품 카테고리                        |
| `price`        | numeric(10,2)          | 상품 가격                            |

조회된 상품은 다음과 같다.

| product_id | product_name | category    |      price |
| ---------: | ------------ | ----------- | ---------: |
|          1 | Laptop       | Electronics | 1200000.00 |
|          2 | Keyboard     | Electronics |   80000.00 |
|          3 | Desk Chair   | Furniture   |  150000.00 |
|          4 | Monitor      | Electronics |  300000.00 |

#### 해석

* `product_id`는 상품을 구분하는 기본 키다.
* `orders.product_id`와 `products.product_id`를 기준으로 주문 데이터와 상품 데이터를 연결할 수 있다.
* `products` 테이블에는 상품 가격이 있으므로, 주문 수량과 가격을 곱해 주문별 매출을 계산할 수 있다.
* 상품 카테고리는 `Electronics`, `Furniture`로 구분되어 있으며, 이후 카테고리별 매출 분석에 활용할 수 있다.

#### 추가 학습 메모

* `price` 컬럼의 자료형은 `numeric(10,2)`이다.
* `numeric(10,2)`는 전체 10자리 숫자 중 소수점 이하 2자리까지 저장할 수 있다는 의미다.
* 금액 데이터는 정밀도가 중요하므로 실무에서는 `integer`보다 `numeric` 계열을 사용하는 경우가 많다.
### 05. 주문별 매출 계산

주문 수량과 상품 가격을 이용해 주문 1건당 매출을 계산한다.

[SQL 실습 열기 →](../02_projects/day02_order_product_analysis/05_order_revenue.sql)

#### 실행 목적

`orders` 테이블에는 주문 수량만 있고 상품 가격은 없다.
따라서 `orders`와 `products`를 `product_id` 기준으로 연결한 뒤, `quantity × price`로 주문별 매출을 계산한다.

#### 핵심 문법

* `INNER JOIN`

  * 두 테이블에서 연결 조건이 일치하는 행만 결합한다.

* `ON o.product_id = p.product_id`

  * 주문 테이블의 상품 ID와 상품 테이블의 상품 ID를 연결 조건으로 사용한다.

* `o.quantity * p.price`

  * 주문 수량과 상품 가격을 곱해 주문별 매출을 계산한다.

* `AS order_amount`

  * 계산 결과에 `order_amount`라는 별칭을 붙인다.
